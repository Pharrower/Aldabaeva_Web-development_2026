## 1. Введение и цели

### 1.1. Краткое описание системы

Информационная система для приюта «Доброе сердце» предназначена для автоматизации учета питомцев и управления складскими запасами. Система предоставляет инструменты для обработки заявок на адопцию, регистрации волонтеров и фиксации пожертвований, обеспечивая прозрачность деятельности приюта.

### 1.2. Цели архитектуры

| Цель                           | Описание                                                                              |
| :----------------------------- | :----------------------------------------------------------------------------------- |
| **Разделение ответственности** | Чёткое разделение UI (Thymeleaf), бизнес-логики и доступа к данным через слои PCMEF. |
| **Тестируемость**              | Изоляция бизнес-логики в Service-слое для покрытия Unit-тестами.                     |
| **Масштабируемость**           | Модульная структура позволяет добавлять новые функции без модификации ядра.          |
| **Поддерживаемость**           | Использование стандартов Spring Boot упрощает сопровождение кода.                    |

### 1.3. Стейкхолдеры

| Стейкхолдер   | Интересы                                                  |
| :------------ | :-------------------------------------------------------- |
| Разработчики  | Понятная структура компонентов, чистота API.              |
| Администратор | Стабильность работы, удобство управления данными.         |
| Проверяющий   | Соответствие структуры PCMEF и качественная документация. |

## 3. Контекст системы

### 3.1. Бизнес-контекст

Система автоматизирует полный цикл жизни животного в приюте: от поступления до передачи новому владельцу.
*   **Входы:** Анкеты животных, заявки на адопцию, данные о кормах.
*   **Выходы:** Статусы заявок, списки питомцев, отчетность для руководства.
*   **Подробно:** [`00-project-charter/context-diagram.md`](../00-project-charter/context-diagram.md)

### 3.2. Технический контекст

Система работает как веб-приложение. Клиент (браузер) взаимодействует с сервером через HTTP, сервер работает с PostgreSQL через JPA/Hibernate.

## 4. Стратегии

### 4.1. Стратегия декомпозиции

Система спроектирована с использованием паттерна PCMEF, который обеспечивает строгое разделение ответственности и высокую модульность компонентов:
*   **Presentation:** Серверный рендеринг шаблонов (Thymeleaf).
*   **Control:** Обработка HTTP-запросов и маршрутизация (Spring MVC Controllers).
*   **Mediator:** Изолированная бизнес-логика и правила приюта (Spring @Service).
*   **Entity:** Описание доменных моделей (JPA Entities).
*   **Foundation:** Инфраструктурный слой доступа к данным (Spring Data JPA Repositories).

Подробное обоснование выбора паттерна PCMEF:  
[`02-architecture/adr/adr-001-pcmef-pattern.md`](adr/adr-001-pcmef-pattern.md)

### 4.2. Стратегия управления данными

- **Хранилище:** Использование СУБД PostgreSQL как надежного реляционного хранилища для структурированных данных (животные, заявки, ресурсы).

- **ORM:** Интеграция с Spring Data JPA для абстрагирования от низкоуровневых SQL-запросов.

- **Транзакционность:** Использование декларативного управления транзакциями через @Transactional на уровне сервисного слоя (Mediator), что обеспечивает целостность данных при выполнении сложных операций.

Подробное обоснование выбора стека данных:  
[`02-architecture/adr/adr-002-db-and-orm.md`](adr/adr-002-db-and-orm.md)

### 4.3. Стратегия безопасности

Для защиты приложения и разграничения прав доступа реализован комплексный подход:

- **Аутентификация:** Модуль Spring Security с сессионной моделью, настроенный на работу с пользователями из БД.

- **Хеширование:** Безопасное хранение паролей с использованием алгоритма BCrypt.

- **Авторизация:** Управление доступом на уровне ролей (ROLE_USER, ROLE_ADMIN) для разделения прав волонтеров и администраторов.

Подробное обоснование стратегии безопасности:  
[`02-architecture/adr/adr-003-authentication.md`](adr/adr-003-authentication.md)

[`04-detailed-design/sequence-diagrams.md`](../04-detailed-design/sequence-diagrams.md)

Ключевой сценарий «Оформление заявки на адопцию»:
1. Пользователь заполняет форму на UI.
2. Controller валидирует данные.
3. Mediator (Service) создает запись в БД через Foundation (Repository).
4. Пользователь получает подтверждение.
