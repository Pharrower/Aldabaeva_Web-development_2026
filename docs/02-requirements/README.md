# Этап 2: Спецификация и анализ требований к системе (Requirements Engineering)

## Описание этапа
На данном этапе произведен переход от бизнес-моделирования к формированию системных требований к программному продукту. Определены функциональные и нефункциональные границы системы, зафиксированы роли пользователей (акторы) и спроектирована концептуальная модель предметной области (Domain Model) для последующего отображения на реляционную базу данных.

## Содержимое каталога
* [use-case-diagram.md](use-case-diagram.md) — Функциональная диаграмма прецедентов использования системы (Use Case Diagram).
* [use-case-specifications.md](use-case-specifications.md) — Детализированные текстовые спецификации ключевых прецедентов.
* [domain-model.md](domain-model.md) — Доменная концептуальная модель классов предметной области.

## Ограничения и нефункциональные требования
1. **Безопасность:** Разграничение доступа к операциям изменения данных (добавление питомцев, управление складом) только для авторизованных волонтеров.
2. **Надежность:** Двухконтурная валидация данных — мгновенная проверка на стороне клиента и строгий контроль на стороне сервера.
3. **Производительность:** Время асинхронного AJAX-отклика интерфейса при отправке заявки не должно превышать 2 секунд.
