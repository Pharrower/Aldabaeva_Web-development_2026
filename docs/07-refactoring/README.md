# Рефакторинг и архитектурное улучшение

Данная папка содержит результаты глубокого рефакторинга проекта "Приют". На этом этапе основное внимание было уделено приведению кода к промышленным стандартам, улучшению его читаемости, тестируемости и масштабируемости.

## 🎯 Цели рефакторинга
1. **Декомпозиция монолита:** Разделение "Божественного объекта" (`AnimalService`) на специализированные сервисные компоненты согласно принципу единственной ответственности (SRP).
2. **Улучшение архитектуры:** Переход к классической многослойной архитектуре (Controller -> Service -> Repository).
3. **Безопасность и надежность:** Замена небезопасных вызовов данных на работу с `Optional` и внедрение конструкторной инъекции зависимостей.
4. **Устранение "запахов кода":** Избавление от дублирования кода, хардкода статусов и избыточных связей между контроллерами и репозиториями.

## 📂 Структура папки
- `refactoring-plan.md` — стратегия проведения рефакторинга.
- `code-smells.md` — перечень устраненных проблем архитектуры.
- `patterns-applied.md` — описание использованных паттернов проектирования.

## 🛠 Ключевые изменения
- **Сервисный слой:** Выделены `DonationService`, `VolunteerService`, `SupplyService` и `UserService`.
- **Инъекция зависимостей:** Переход от `@Autowired` над полями к внедрению зависимостей через конструкторы (`Constructor Injection`), что делает код более стабильным и удобным для написания Unit-тестов.
- **Инкапсуляция:** Контроллеры теперь взаимодействуют только с сервисами, не имея прямого доступа к репозиториям, что соответствует паттерну Data Mapper и принципу инкапсуляции.

## 🚀 Итог
Проект стал модульным. Теперь добавление новой функциональности (например, новых типов донатов или типов заявок) не требует изменений в существующих базовых сервисах, что значительно упрощает дальнейшую поддержку и развитие системы.
